# immutable-jrmc

Standalone tooling for building a JRiver Media Center AppImage that runs reliably on immutable Linux distributions such as Bazzite, Silverblue, and related Fedora Atomic desktops.

This project exists because the current JRiver Linux packaging model fits traditional mutable Debian and Ubuntu systems better than immutable Fedora-style desktops. The AppImage produced here is a compatibility layer around the existing Linux release, not a replacement for proper upstream support.

## Why This Exists

Immutable distributions are getting more popular because they make upgrades, rollback, and day-to-day system maintenance easier for non-expert users while still working well for gaming, containers, and developer workflows.

JRiver Media Center can work well on Linux, but two classes of problems make the current experience rough on Bazzite and Silverblue:

1. Fedora regressions in JRiver itself.
	Recent JRMC 35 builds can crash on launch on Fedora-based systems in browser-related startup paths. The current workaround in this builder applies a trampoline patch that guards a null delegate pointer in the startup path associated with `CActionWindowArray`.
2. Assumptions that do not fit immutable hosts.
	JRiver expects to live in fixed paths such as `/usr/lib/jriver/Media Center 35` and `/usr/bin/mediacenter35`. On immutable systems, `/usr` is read-only and those assumptions make portable deployment difficult.

This repository packages the existing JRiver Linux install into a relocatable AppImage and adds the minimum runtime fixes needed to make it usable on immutable hosts.

## Project Goals

- Make JRiver usable on immutable Linux systems without asking the user to modify `/usr`.
- Preserve a native local desktop experience with Wayland/Xwayland instead of remoting into another machine or VM.
- Keep the build process reproducible and auditable.
- Document the specific upstream JRiver issues that should eventually make these workarounds unnecessary.

## Non-Goals

- This is not intended to be a permanent downstream fork of JRiver.
- This is not a reimplementation of JRiver packaging from scratch.
- This does not try to make Fedora regressions disappear at the source; it records and works around them until upstream fixes are available.

## Repository Contents

- `jriver-appimage-packager.sh`: the end-to-end AppImage build script.
- `.devcontainer/`: the canonical local build environment for VS Code Dev Containers.
- `README.md`: architecture, motivation, usage, and design notes.

The preload shim source is generated by the packager during the build. It is embedded in the script because the current project is intentionally small and focused on a single build artifact.

## High-Level Architecture

The builder follows a simple pipeline:

1. Install or refresh JRiver inside a Debian-based build environment, typically a `jriver` distrobox.
2. Copy the installed JRiver payload into an AppDir.
3. Apply runtime fixes needed for relocatable execution.
4. Bundle the dependent shared libraries required to run outside the original container filesystem.
5. Create an AppImage from the resulting AppDir.
6. Emit manifests and optional baselines so a known-good build can be preserved and compared later.

The resulting AppImage is designed to run directly on the host, with an explicit preference for a direct relocatable runtime path over namespace tricks.

## Runtime Design Choices

### 1. Direct relocatable execution is preferred

The AppImage launcher prefers a direct execution path because immutable hosts make namespace-based fake `/usr` trees unreliable. On Bazzite and similar systems, `/usr` is read-only, so approaches that depend on synthesizing `/usr/lib/jriver` are fragile.

The launcher still keeps compatibility fallbacks:

- direct mode
- `bwrap` when viable
- `proot` as a fallback compatibility path

This ordering is intentional. Direct execution is simpler, faster, easier to debug, and better aligned with how immutable desktops behave.

### 2. JRiver path assumptions are redirected instead of trusted

JRiver currently behaves like a fixed-layout application. It expects binaries, resources, and helper processes to exist under paths such as:

- `/usr/lib/jriver/Media Center 35`
- `/usr/bin/mediacenter35`
- `/usr/lib/jriver-runtime`

The generated preload shim intercepts runtime calls and redirects those paths into the AppDir. This includes:

- file opening and probing
- stat-family lookups
- `dlopen`
- path canonicalization
- helper process launch APIs such as `execve`, `posix_spawn`, `system`, and `popen`

This is the main reason the AppImage can run cleanly on immutable hosts even though JRiver itself is not fully relocatable.

### 3. RPATHs are rewritten to make the payload self-contained

The builder uses `patchelf` to rewrite ELF RPATHs so the copied binaries and shared objects resolve from the AppDir instead of assuming a distro-installed runtime layout.

Without this step, direct execution fails much earlier because bundled libraries are not resolved relative to the AppImage payload.

### 4. Startup patching is explicit and verified

The builder applies a binary patch to the copied `mediacenter35.vendor` launcher, and to the copied payload launcher when needed. The patch is byte-verified before and after writing so the build fails loudly if JRiver changes in a way that invalidates the expected patch offsets.

This is not cosmetic. The patch works around a real startup crash in the current Linux release.

### 5. Full builds publish AppDir atomically

The builder assembles into `AppDir.work` and only replaces the public `AppDir` after a successful build. This avoids leaving behind a half-built runtime tree if the build is interrupted.

### 6. Build artifacts are documented, not just produced

The builder emits:

- build logs
- AppDir tree manifests
- rewritten path audits
- bundled library manifests
- artifact metadata including checksum and size
- optional baseline snapshots for known-good releases

That makes regression tracking much easier when JRiver changes upstream.

## Motivation For Using An AppImage

AppImage is a good fit here for a few reasons:

- It gives users a single portable artifact instead of a custom distrobox workflow.
- It avoids requiring mutable system paths on the host.
- It is easy to test across Bazzite, Silverblue, and other immutable desktops.
- It allows the payload, runtime fixes, and libraries to travel together.
- It preserves a normal local desktop launch model without remoting.

This project is not claiming AppImage is the ideal upstream packaging format for JRiver. It is simply the most practical way to bundle the current Linux build and its workarounds into something users can actually run on immutable systems today.

## Known Upstream Issues This Builder Works Around

### Fedora startup crash

The current JRMC 35 Linux build has a startup crash in the browser-related UI path on Fedora-based systems. Investigation isolated the problem to a null delegate pointer dereference associated with `CActionWindowArray + 0x238`.

The current workaround is a trampoline patch that adds the missing guard before the dereference. Upstream should fix this in JRiver itself so binary patching is no longer necessary.

For regression testing, the builder also supports a reproducible opt-out at build time. Set `JRIVER_APPIMAGE_DISABLE_STARTUP_GUARD=1` when invoking the packager to produce an artifact without the workaround so the original crash path can be reproduced on affected Fedora-based hosts.

### Fixed-path install assumptions

JRiver assumes fixed host paths under `/usr/lib/jriver` and `/usr/bin`. Those assumptions are awkward even for portable packaging and directly conflict with immutable systems where `/usr` is not writable.

Upstream should instead:

- derive the install root dynamically at runtime
- resolve helpers and resources relative to that install root
- default writable/configurable paths to user-owned locations when appropriate

### Fragmented helper/resource resolution

The preload shim had to intercept multiple different APIs to make helper launches and resource lookups work consistently. That implies JRiver is not using a single unified runtime root strategy internally.

Upstream would benefit from centralizing how:

- helpers are launched
- libraries are located
- resource files are opened
- browser-related subprocesses are started

## Build Environment

Run the build inside the `jriver` distrobox or a compatible Debian-based environment with:

- `bash`
- `curl`
- `python3`
- `gcc`
- `patchelf`
- `proot`
- `rsync`
- `squashfs-tools`
- `desktop-file-utils`
- `file`
- `binutils`

The script can install most prerequisites itself unless `--skip-prereqs` is used.

For local testing on Bazzite-dx, the preferred approach is to use the repo's
Ubuntu 24.04 container scaffold rather than trying to build directly on the
immutable host.

## Usage

Basic build:

```bash
bash jriver-appimage-packager.sh
```

The builder currently pins the JRiver install step to `35.0.54` and installs it
from the official source DEB so the startup guard patch is applied against a
known compatible launcher build even after newer repo packages are published.
The script also logs `dpkg-query -W mediacenter35` during layout detection so CI
runs show the actual installed package version.

Build and preserve a known-good baseline:

```bash
bash jriver-appimage-packager.sh --skip-prereqs --baseline-label 2026-03-20-working
```

Build an artifact that intentionally omits the startup crash workaround:

```bash
JRIVER_APPIMAGE_DISABLE_STARTUP_GUARD=1 \
	bash jriver-appimage-packager.sh --skip-prereqs --baseline-label reproduce-cactionwindowarray-crash
```

Override the build root and export a copy of the finished artifact:

```bash
bash jriver-appimage-packager.sh \
  --build-root "$HOME/.local/state/jriver-appimage" \
  --export-dir "$PWD/output"
```

## VS Code Dev Container

The repository now includes a first-class dev container configuration at
`.devcontainer/devcontainer.json` for Bazzite-dx and other container-first
Linux hosts.

Design choices:


- It uses a dedicated `.devcontainer/Dockerfile` based on the official
	`mcr.microsoft.com/devcontainers/base:ubuntu` image as the single supported
	local container environment.
- It relies on the built-in non-root `vscode` user from the devcontainers base
	image, so VS Code can use its normal UID/GID alignment mechanism on Linux
	without any custom `useradd` or `usermod` logic in the devcontainer Dockerfile.
- It keeps a persistent named volume for `/home/vscode` so shell history,
	JRiver bootstrap state, and editor-local container state survive rebuilds.
- It keeps `BUILD_ROOT` and `XDG_STATE_HOME` under `./.local-build/devcontainer/`
	so local logs and state remain visible from the workspace.

Recommended setup on Bazzite-dx:

1. Install the VS Code Dev Containers extension.
2. Ensure VS Code is configured to work with your container runtime. Podman is
	 the recommended default on Bazzite-dx.
3. Open the repository in VS Code and use `Dev Containers: Reopen in Container`.
4. Run one of the workspace tasks:
	 `Build AppImage`, `Build AppImage (Guard Disabled)`, or `Clean Local Build State`.

This is the supported local workflow. The repository no longer carries a
separate host-managed `podman` or `docker` wrapper path.

Local scaffold directories:

- `./dist/`: exported AppImages and copied manifests
- `./.local-build/devcontainer/`: workspace-visible build root and XDG state
	used by the devcontainer tasks

Recommended local loop:

1. Reopen the repository in the devcontainer.
2. Run `Build AppImage` or `Build AppImage (Guard Disabled)` from the workspace tasks.
3. Inspect `./dist/manifests/` and the step logs under `./.local-build/devcontainer/` if the build fails.
4. Push only after the task run passes.

## Outputs

Default build root:

- non-root: `$XDG_STATE_HOME/jriver-appimage` or `$HOME/.local/state/jriver-appimage`
- root: `/var/lib/jriver-appimage`

Notable output directories:

- `output/`: built AppImage artifacts
- `logs/`: step-by-step build logs and stderr captures
- `manifests/`: build metadata, path audits, and dependency manifests
- `baselines/`: optional preserved known-good snapshots
- `AppDir/`: last successfully published AppDir
- `AppDir.work/`: temporary in-progress AppDir during a build

## Current State

The current builder has produced known-good baselines where:

- JRiver launches successfully
- audio playback works
- video playback works
- direct host execution is preferred over namespace-based compatibility layers

That does not mean the upstream Linux packaging issues are solved. It means the current set of workarounds is sufficient to make JRiver usable on immutable desktops.

## Why This Should Eventually Go Away

This repository is best understood as a practical compatibility guide.

The ideal long-term outcome is that JRiver upstream:

- fixes the Linux startup crash
- removes hardcoded `/usr` assumptions
- supports relocatable or immutable-friendly installation layouts
- behaves consistently on Fedora-based systems without downstream patching

If that happens, the need for this AppImage builder becomes much smaller.

## Forum Summary

If you are looking at this repository because of a community post: that is the intent.
